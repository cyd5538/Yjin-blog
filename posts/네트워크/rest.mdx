# REST(Representational State Transfer)
## REST 정의
"Representational State Transfer" 의 약자로, 
자원을 이름(자원의 표현)으로 구분해 해당 자원의 상태(정보)를 주고 받는 모든 것을 의미합니다.
즉, <u>**자원(resource)의 표현(representation)에 의한 상태 전달을 뜻합니다.**</u>
* 자원: 모든 것을 자원으로 간주(웹 페이지, 데이터, 이미지 등)
* 표현: 자원을 다양한 형식의 표현으로 나타낼 수 있다
* 상태 전달: 데이터가 요청되는 시점에 클라이언트에게 적절한 상태 전달을 제공한다.(JSON 또는 XML)

REST는 자원의 이름을 구분하고 해당 자원의 상태를 주고 받는 웹의 장점을 최대한 활용할 수 있는 아키텍쳐 스타일로, 
Client와 Server 간의 통신 방식 중 하나다.
## REST 개념
**HTTP URI(Uniform Resource Identifier)를 통해 자원(Resource)을 명시하고, 
HTTP Method(POST, GET, PUT, DELETE)를 통해 해당 자원에 대한 CRUD Operation을 적용하는 것을 의미한다.**

> **URL vs URI**
> * URI (Uniform Resource Identifier): URI는 인터넷 상에서 자원을 식별하는 일반적인 개념
> * URL (Uniform Resource Locator): URL은 URI의 하위 개념 중 하나로 특정한 자원이 어디에 위치하고 어떻게 접근할 수 있는지를 나타내는 식별자

> **CRUD Operation**
> - Create : 생성(POST)
> - Read : 조회(GET)
> - Update : 수정(PUT)
> - Delete : 삭제(DELETE)

## REST의 특징
1. **Server-Client 구조**: REST는 자원을 제공하는 서버와 자원을 요청하는 클라이언트 간의 구조를 가진다.
<u>서버는 API를 제공하고 비즈니스 로직 처리를 담당하며, 클라이언트는 요청을 보내고 응답을 받는다.</u>

2. **무상태**: REST는 무상태(Stateless) 프로토콜로, 클라이언트의 상태를 서버에 저장하지 않는다. <u>각 요청은 완전히 독립적으로 처리</u>되며, 서버는 이전 요청과 다음 요청 간의 연관성을 갖지 않는다.

3. **캐시 처리 가능**: HTTP의 캐싱 기능을 활용하여 응답을 캐시할 수 있다. ➡ 성능 향상과 서버 자원 이용률 개선이 가능

4. **계층화**: REST 서버는 다중 계층으로 구성될 수 있으며, 클라이언트는 API 서버만 호출 계층화를 통해 보안, 로드 밸런싱, 암호화, 인증 등을 추가 할 수 있다.

5. **인터페이스 일관성**: REST는 통일된 인터페이스를 제공하며, URI를 통해 자원을 일관되고 한정적인 방식으로 조작한다. 따라서  다양한 플랫폼에서 사용 가능하며, 언어나 기술에 종속되지 않는다.


# REST API
## REST API 란
* REST 원칙에 따라 설계된 웹 API의 한 형태다
* OpenAPI(누구나 사용할 수 있는 공개된 API) 는 대부분 REST API를 제공

## REST 설계 기본 규칙
1. URI는 정보의 자원을 표현해야 한다.
  - URI만 보고 어떤 요청을 하는지 파악이 가능해야 한다.
2. 자원에 대한 행위는 HTTP Method(GET, PUT, POST, DELETE) 로 표현한다.
  - 행위는 URI에 들어가면 안된다 (/user/delete/1 ❌)

REST API의 설계 규칙
- 슬래시 구분자(/)는 계층 관계를 나타내는데 사용하자.
- 가독성을 높일 땐 밑줄 _을 사용하지 말고 하이픈(-)을 사용하자.
- URI 경로에는 소문자를 사용하자.
- URI 마지막 문자로 슬래시를 포함하지 않는다
- 파일 확장자는 URI에 포함하지 않는다.

REST API 설계 예시

| HTTP 메서드 | 엔드포인트 (URI)           | 설명                                  |
| ------------ | --------------------------- | ------------------------------------- |
| GET          | `/users`                    | 모든 사용자 목록 조회                |
| GET          | `/users/{userId}`           | 특정 사용자 조회                      |
| POST         | `/users`                    | 새로운 사용자 생성                    |
| DELETE       | `/users/{userId}`           | 특정 사용자 삭제     |

## RESTful 하다?
REST를 사용했다 하여 모두가 RESTful 한 것이 아니다. <u>REST API의 설계 규칙을 잘 따르시스템을 RESTful하다라고 말한다.</u>

**RESTful하지 못한 경우**
* CRUD 기능을 모두 POST 로만 처리하는 API
* route에 resource, id 외의 정보가 들어가는 경우
