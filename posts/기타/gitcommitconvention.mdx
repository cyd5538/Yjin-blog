# Git Commit Convention이란 ✍
깃 커밋 컨벤션은 프로젝트의 협업과 버전 관리를 원활하게 하기 위해 정해진 규칙에 따라 커밋 메시지를 작성하는 방법입니다.

# 커밋 메시지 구조 ✏
커밋 메시지는 크게 제목(Subject), 본문(Body), 풋터(Footer)로 구분됩니다.

```
subject -> 제목
body -> 본문
footer -> 푸터
```
## subject
* 커밋의 주요 내용을 요약하여 한 줄로 작성합니다.
* 첫 글자는 대문자로 시작하며, 마침표를 사용하지 않습니다.
* 최대 50자 이내로 작성하는 것이 좋습니다.
* 예시: "feat: 로그인 기능 구현"
## body
** 커밋의 상세한 내용을 작성합니다.
** 왜 이 작업이 수행되었는지, 어떤 변경 사항이 있는지 등을 자세하게 설명합니다.
** 가독성을 위해 72자 정도마다 줄을 바꿔줍니다.
## footer
* 커밋과 관련된 이슈 번호나 특별한 명령을 작성합니다.
* 주로 이슈 번호를 참조하거나, 커밋에 대한 특별한 액션을 지정할 때 사용합니다.

> 커밋 메시지 작성 시 유의할 점 ✔
> * 메시지는 명확하고 간결하게 작성하세요.
> * 오타와 띄어쓰기에 신경 쓰세요!

# 커밋 타입 🤞

|커밋 타입|의미|예시|
|---|---|---|
|feat	|새로운 기능 추가 (Feature)	|"feat: 사용자 로그인 기능 구현"|
|fix	|버그 수정 (Bugfix)	|"fix: 로그인 버튼 오류 수정"|
|chore	|빌드 및 환경 설정 변경 (Chore)	|"chore: 프로젝트 빌드 설정 업데이트"|
|refactor|	코드 리팩토링 (Refactor)	|"refactor: 중복 코드 제거 및 변수명 변경"|
|docs	|문서 업데이트 (Documentation)	|"docs: API 사용법 업데이트"|
|style	|코드 스타일 변경 (Styling)	|"style: 코드 정렬 및 들여쓰기 수정"|
|test	|테스트 코드 추가 또는 수정 (Testing)	|"test: 사용자 등록 기능에 대한 유닛 테스트 추가"|
|perf	|성능 개선 (Performance)	|"perf: 데이터베이스 쿼리 최적화"|
|build	|빌드 관련 변경 (Build)	|"build: 빌드 스크립트 업데이트"|
|ci	CI/CD |설정 변경 (Continuous |Integration / Continuous Deployment)	"ci: 테스트 자동화를 위한 CI 설정 추가"|

# 커밋 예시 ⭐
```
feat: 사용자 로그인 기능 구현

사용자의 이메일과 비밀번호를 입력하여 로그인하는 기능을 추가했습니다.
비밀번호가 일치하지 않을 경우 알림 메시지를 표시하고,
정상적으로 로그인되면 환영 메시지를 보여줍니다.

Closes #123
```
